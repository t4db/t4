01 / DURABILITY

Survive anything.

Every write is streamed to S3 before it's acknowledged. When a node loses its disk — T4 restores from S3 automatically on next boot. No runbook. No 3am pages. No data loss.

02 / COMPATIBILITY

Drop in, don't rip out.

T4 speaks the etcd v3 gRPC protocol. Your existing clients, tooling, and Helm charts work unchanged. Swap the endpoint, keep everything else. Zero migration scripts.

03 / SIMPLICITY

Ship it inside your app.

One t4.Open() call. No sidecar daemon, no extra process. Embed the store directly in your Go binary. Local reads: zero network, zero latency.

How it works

S3 as the source
of truth.

T4 writes every change to a local WAL and streams it to S3. Leader election uses S3 atomic conditional PUTs — no Raft, no ZooKeeper, no cluster membership config.

Any node can join or leave at any time. A replacement node fetches the latest S3 checkpoint, replays recent WAL segments, and is serving traffic within seconds — automatically.

For multi-writer workloads, T4's S3-based consensus scales better than Raft under contention. Single-writer performance is slower. See the benchmarks.

// No daemon. No sidecar. One call.
node, _ := t4.Open(t4.Config{

    ObjectStore: s3Store,
})
defer node.Close()

// Write

// Read — zero network, zero latency
kv, _ := node.Get("/cfg/timeout")

// Watch any prefix
ch, _ := node.Watch(ctx, "/cfg/", 0)
Comparison

T4 vs etcd

Same wire protocol. Completely different durability model. T4 trades Raft's write-concurrency advantages for disk-loss immunity and S3-native recovery.
